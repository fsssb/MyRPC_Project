# MyRPCProject — 项目说明

## 1. 项目概览

**MyRPCProject** 是一个用 **C++17** 实现的 **高并发网络 RPC 示例**（非生产级框架），演示 **Main–Sub Reactor**、**长度头分帧**、**模拟 AI 网关** 与 **指标/优雅关闭** 的完整闭环。

| 项 | 说明 |
|----|------|
| 定位 | 学习/简历/面试可演示的「自研 Reactor + 协议 + 业务编排」小工程 |
| 主程序 | `rpc_demo`：监听 `0.0.0.0:12345`，可配置 Sub Reactor 工作线程数 |
| 多路复用 | **Linux**：`epoll` **ET**；**macOS**：`poll`（同一套 `EventLoop` / `Channel` 抽象） |
| 容器 | 推荐在 **Linux 容器** 中跑以验证 **Epoll** 与 ET 下读写行为，与本机 `poll` 对照 |

---

## 2. 技术栈与构建要求

- **语言**：C++17（`CMAKE_CXX_STANDARD 17`）
- **构建**：CMake ≥ 3.16
- **线程库**：Pthread（`Threads::Threads`）
- **无第三方网络库**：自实现 `EventLoop`、`Poller`、`TcpConnection` 等

**可选工具**：Python 3（`scripts/bench.py`、`scripts/rpc_e2e_client.py`、集成验收脚本 Docker 内/宿主机用）

## 4. 架构说明

### 4.1 Main–Sub Reactor

- **主 `EventLoop`**：负责 `Acceptor` 接受新连接、定时器（如指标、优雅退出检测）、**通过 `ThreadPool` 经主循环投递业务任务**（`queueInLoop`），避免无锁 MPSC 队列在并发下的不确定性。
- **子 `EventLoop`（I/O 线程）**：`TcpConnection` 挂在子循环上，负责读/写、编解码、回调上抛到主循环执行业务。
- **`ThreadPool`**：维护多个子 `EventLoop` 的线程；`getNextLoop()` 对连接做轮询绑定。

### 4.2 核心组件关系（简述）

- **`Poller` / `EpollPoller` / `PollPoller`**：跨平台多路复用；Linux 为 **ET**，读侧一般需读到 `EAGAIN`。
- **`Channel`**：将 fd 事件绑定到 `EventLoop` 与回调。
- **`TcpConnection` + `Codec`**：解决粘包/半包，完整帧后组 `Message` 回调业务。
- **`TcpServer` + `Acceptor`**：监听与分发新连接到子 `EventLoop`；**定时**检查空闲连接并在超时后关闭（默认 30s，见 `TcpServer` 实现）。
- **`AIService` + `TaskCoordinator`**：模拟 AI 意图/推理与编排耗时（可配置工作线程等）。
- **`Metrics`**：请求计数、延迟分桶、Prometheus 风格文本等（`main` 中周期打印）。
- **`EventLoop` 唤醒管道**：`queueInLoop` 跨线程时 `write` 唤醒；**Linux 下管道两端为 `O_NONBLOCK`**，与 **ET** 下「读到 `EAGAIN`」的约定一致，避免子循环卡在阻塞读上无法处理挂起的 functor。

## 5. 网络协议

- **分帧形式**：`[4 字节大端无符号整型，表示 body 长度][body 原始字节]`
- 服务端仅在对齐到完整一帧后进入业务；响应同样按此格式回写。

---

## 6. 运行与参数

```bash
cmake -B build && cmake --build build
./build/rpc_demo <SubReactor 线程数>
```

- 第一个参数为 **Sub Reactor 工作线程数**（`main` 里还会据此推导 AI 侧 worker 数等，见 `main.cpp`）。
- 未传或非法时，按代码内默认处理（如默认 `4`）。

默认监听 **12345**；`Dockerfile` 的 `EXPOSE` 与 `CMD` 与此一致，可按需改代码或 `docker run -p` 映射。

---

## 7. Docker（推荐用于 Linux + Epoll 验证）

```bash
docker build -t myrpc-epoll .
docker run --rm -p 12345:12345 myrpc-epoll
```

- 构建前建议确认构建上下文在**项目根目录**；代码变更后**需要重新构建**镜像再测。
- 拉取/代理、ECR 与 Hub 切换、**zsh 与 e2e 脚本** 等，以 **`README.md`** 中「用 Docker 在 Mac 上跑」与「验收」小节的**实操排障**为准，本文不重复长命令与故障列表。

**一键验收**（需本机已安装且可用 Docker，且宿主机可访问映射端口）：

```bash
bash ./scripts/acceptance.sh myrpc-epoll myrpc-e2e 12345
```

成功时依次出现 **IN_CONTAINER: OK**、**HOST: OK** 与短压测统计；`exit 0` 即视为本脚本定义下的验收通过。

> **注意**：在无 Docker 或 `127.0.0.1` 与宿主机网络隔离的远程沙箱中执行第 4 步「宿主机自测」可能失败，属**环境假阳性**；**第 3 步容器内 e2e** 才是「服务逻辑与镜像是否可用」的硬指标。

## 8. 压测

```bash
python3 scripts/bench.py --host 127.0.0.1 --port 12345 --concurrency 5 --duration 2
```

业务层**刻意**模拟较重型 AI 编排与延迟，高并发下 QPS 受业务侧限制是**设计使然**，不单独代表网络栈极限。详细示例输出见 `README.md` 压测小节。

## 9. 设计要点与排障提示（精要）

1. **ET + 唤醒管道**：在 Linux/ET 上，wakeup 读端需非阻塞、一次尽量读空直到 `EAGAIN`；若管道为阻塞可能令子 `EventLoop` 无法推进 `doPendingFunctors()`，表现为连接/回调「永远不执行」、客户端等不到响应头。
2. **业务任务经主 `EventLoop`**：通过 `queueInLoop` 保证投递语义清晰；`inferAsync` 为异步、不应长时间占满主循环。
3. **自动化验收时容器 `--rm` 与失败停容器**：若需日志，请在本机用 `docker logs` 在失败**前**抓取，或改脚本/手动起不退出的容器，便于 `docker logs` 与进容器看 `/app/rpc.log`。

更细的命令、代理与**验收报告**撰写约定，以仓库内 `README.md` 与 `验收报告.md` 为准。
